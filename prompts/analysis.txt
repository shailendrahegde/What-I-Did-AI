RULE 1 — GOAL GROUPING (most important rule)
═══════════════════════════════════════════

Group work into BUSINESS GOALS. A goal = the high-level outcome accomplished, not the technical steps.

IRON RULE: Everything in the same session that touches the same system is ONE GOAL.
Iterating, refining, fixing bugs, adding features — all tasks within the original goal, not separate goals.

DEFAULT RULE: When in doubt, MERGE. Too few goals is better than too many.

The only valid reason to create a new goal is: work on a COMPLETELY DIFFERENT subject in a DIFFERENT
project/session with ZERO shared files or dependencies.

ROOT/HOME FOLDER EXCEPTION: If a session's PROJECT is a user home directory (e.g. "jsmith",
"username", or a path like /Users/name or C:\Users\name), the user was likely doing ad-hoc work
that does NOT belong to a single project. In this case:
- Do NOT merge all sessions from the same home folder into one giant goal.
- Only group sessions together if they share CLEAR evidence of the same project: same files
  modified, same topic/domain, same repository, or explicit continuation of prior work.
- Separate ad-hoc tasks (e.g. a quick research question, installing a tool, answering a question)
  should each be their own small goal or grouped only with closely related tasks.
- Look at the actual CONTENT and FILES in each session, not just the project path.

GOOD GOAL TITLES (business outcome, verb-first, INCLUDE the project/repo/tool name):
  "Built a daily AI activity analytics tool with branded HTML reports"

RULE 3 — EFFORT ESTIMATES (research-calibrated)
═══════════════════════════════════════════

human_hours = what a skilled professional would need WITHOUT AI assistance.
Estimate realistically — what would an expert actually bill for this work?

RESEARCH FOUNDATION:
- LLM-assisted work completes 1.4–4× faster than unassisted work (Cambon et al. 2023, Peng et al. 2023)
- The human EQUIVALENT is what matters: reverse the speedup to get what it would take WITHOUT AI
- No single metric captures effort — use MULTIPLE signals together (SPACE framework, Forsgren 2021)
- Empirical OLS regression on 48 days of sessions (R²≈0.40) identified THREE independent signals:
    1. Conversation turns  — logarithmic: early turns count most, later turns have diminishing returns
    2. Logic code lines    — only .py/.ts/.go/.rs/.java/.sh and similar; HTML/CSS/JSON/MD excluded
    3. Read/search calls   — file reads + grep/glob/search; captures investigation with no code output
- Adding complexity multipliers on top degraded accuracy — the log curves already handle diminishing returns
- Boilerplate lines (HTML/CSS/JSON/MD/YAML) have r=−0.14 correlation with effort; logic lines r=+0.41

Each session includes a SIGNALS block. USE IT to calibrate — do not ignore these numbers.

STEP 1 — Classify the task type using the SIGNALS block:
  Mechanical (install, deploy, git push, config)       → 0.25–0.5h regardless of other signals
  Research/Analysis (read-heavy tools, low edits)      → turns_h + reads_h; lines_h ≈ 0
  Implementation (edit-heavy, logic lines added)        → turns_h + lines_h (logic only) + reads_h
  Debugging (run-heavy, high iteration)                 → turns_h + reads_h; the iteration is already in turns
  Complex multi-step (balanced reads+edits+runs)        → all three terms contribute

STEP 2 — Estimate each of the three additive components:

  TURNS_H — Conversation depth (substantive turns only — exclude 'yes', 'commit', 'ok', single-digit replies):
    Formula: max(0, −0.15 + 0.67 × ln(turns + 1))
    Reference table:
      3 turns → 0.75h    |  8 turns → 1.2h   |  15 turns → 1.6h
      25 turns → 1.9h    |  40 turns → 2.2h   |  60 turns → 2.5h
      80 turns → 2.7h    |  100 turns → 2.8h  |  150 turns → 3.1h
    The log curve captures diminishing returns: going from 5→15 turns adds more effort than 85→95 turns,
    because early turns drive decisions while later turns are refinements.

  LINES_H — Logic code written (ONLY logic code files — NOT boilerplate):
    INCLUDE: .py .js .ts .jsx .tsx .go .rs .java .cs .cpp .c .sh .bash .rb .sql .kt .swift and similar
    EXCLUDE: .html .css .json .md .yaml .yml .csv .xml .toml .lock and all other markup/data/template files
    AI generates boilerplate cheaply; a human expert would write a template, not hand-author every line.
    Formula: 0.40 × log₂(logic_lines ÷ 100 + 1)
    Reference table:
      50 lines → +0.20h   |  100 lines → +0.40h  |  200 lines → +0.63h
      500 lines → +1.03h  |  1000 lines → +1.33h  |  3000 lines → +1.68h

  READS_H — Investigation and analysis (file reads + grep/glob/search tool calls):
    This term captures research, exploration, and debugging sessions where the work is understanding,
    not code output. A session with 50 file reads and searches required skilled navigation and judgment
    even if it produced few or no code lines. (Vaithilingam 2022, SPACE Activity dimension, Alaswad 2026)
    Formula: 0.10 × log₂(read_calls + 1)
    Reference table:
      5 reads → +0.26h  |  10 → +0.35h  |  20 → +0.44h  |  50 → +0.57h  |  100 → +0.67h

  TOTAL = turns_h + lines_h + reads_h
  (minimum 0.25h, rounded to nearest 0.25h)

  These are CALIBRATION ANCHORS. Your semantic judgment about what was actually accomplished,
  the decisions made, and the business value delivered should override the formula when warranted.
  The formula captures the average — you are estimating THIS specific piece of work.

  8–16h    — Large: build a full system from scratch, multi-day design-implement-test cycle
