RULE 1 — GOAL GROUPING (most important rule)
═══════════════════════════════════════════

Group work into BUSINESS GOALS. A goal = the high-level outcome accomplished, not the technical steps.

IRON RULE: Everything in the same session that touches the same system is ONE GOAL.
Iterating, refining, fixing bugs, adding features — all tasks within the original goal, not separate goals.

DEFAULT RULE: When in doubt, MERGE. Too few goals is better than too many.

The only valid reason to create a new goal is: work on a COMPLETELY DIFFERENT subject in a DIFFERENT
project/session with ZERO shared files or dependencies.

ROOT/HOME FOLDER EXCEPTION: If a session's PROJECT is a user home directory (e.g. "jsmith",
"username", or a path like /Users/name or C:\Users\name), the user was likely doing ad-hoc work
that does NOT belong to a single project. In this case:
- Do NOT merge all sessions from the same home folder into one giant goal.
- Only group sessions together if they share CLEAR evidence of the same project: same files
  modified, same topic/domain, same repository, or explicit continuation of prior work.
- Separate ad-hoc tasks (e.g. a quick research question, installing a tool, answering a question)
  should each be their own small goal or grouped only with closely related tasks.
- Look at the actual CONTENT and FILES in each session, not just the project path.

GOOD GOAL TITLES (business outcome, verb-first, INCLUDE the project/repo/tool name):
  "Built a daily AI activity analytics tool with branded HTML reports"

RULE 3 — EFFORT ESTIMATES (research-calibrated)
═══════════════════════════════════════════

human_hours = what a skilled professional would need WITHOUT AI assistance.
Estimate realistically — what would an expert actually bill for this work?

RESEARCH FOUNDATION:
- LLM-assisted work completes 1.4–4× faster than unassisted work (Cambon et al. 2023, Peng et al. 2023)
- The human EQUIVALENT is what matters: start from the speedup and reverse it to estimate without AI
- If a professional was actively engaged for 90 minutes directing AI, the human equivalent is 2–6h
  (1.4–4× range). Use the nature of the work to pick within that range — routine tasks → lower end,
  design and decision-heavy work → upper end.
- No single metric captures effort — use MULTIPLE signals together (SPACE framework, Forsgren 2021)
- Four signals independently explain AI-assisted effort (OLS regression, 48 days):
    1. Conversation turns  — each substantive turn represents real human thinking (framing, deciding, reviewing)
    2. Logic code lines    — only .py/.ts/.go/.rs/.java/.sh and similar; HTML/CSS/JSON/MD excluded
    3. Read/search calls   — file reads + grep/glob/search; captures investigation with no code output
    4. Tool invocations    — total tool calls; captures execution work (browser, commands, images, documents)
       where lines_h ≈ 0 (non-coding tasks). Low coefficient (0.07) avoids double-counting coding work.
- Premium requests serve as fallback interaction signal when turn data is unavailable (older sessions)
- Boilerplate lines (HTML/CSS/JSON/MD/YAML) correlate NEGATIVELY with effort (r=−0.14); logic lines r=+0.41

Each session includes a SIGNALS block. USE IT to calibrate — do not ignore these numbers.

THE FORMULA IS A FLOOR, NOT A CEILING.
The deterministic formula (interaction_h + lines_h + reads_h + tools_h) gives the minimum plausible
estimate based on mechanical counts. Your semantic judgment about the QUALITY of the work — what
decisions were made, what expertise was required, what would an expert firm bill — should typically
result in an estimate AT OR ABOVE the formula floor, not below it. The formula cannot see business
value; you can.

STEP 1 — Classify the task type using the SIGNALS block:
  Mechanical (install, deploy, git push, config)       → 0.25–0.5h regardless of other signals
  Research/Analysis (read-heavy tools, low edits)      → active_minutes × 2–3 is a good anchor
  Implementation (edit-heavy, logic lines added)        → logic lines ÷ 100 LoC/hr + research overhead
  Debugging (run-heavy, high iteration)                 → base + 50% premium for rework and validation
  Non-coding work (browser, docs, images, no code out) → tool_invocations is the primary signal
  Complex multi-step (balanced reads+edits+runs)        → use highest signal as floor, apply judgment upward

STEP 2 — Estimate from multiple signals and use the HIGHEST:

  ACTIVE TIME (primary real-world anchor):
    active_minutes × 2 = conservative human-equivalent (1.4× research lower bound)
    active_minutes × 4 = generous human-equivalent (upper research bound, Cambon 2023)
    Use × 2–3 for routine/mechanical work. Use × 3–4 for design, debugging, research, decision-making.
    Example: 90 min active on a feature design → 3–4h human equivalent is appropriate.

  CONVERSATION TURNS (collaboration depth — substantive turns only, exclude 'yes'/'commit'/'ok'):
    Each substantive turn ≈ 5–8 min of expert thinking (framing, reviewing output, deciding next steps).
    1–3 = 0.25h  |  4–8 = 0.75h  |  9–15 = 1.5h  |  16–30 = 2.5–3h
    31–60 = 3–5h  |  61–100 = 5–8h  |  100+ = 8–12h
    Heavy iteration (many turns on same problem) signals complexity — estimate toward the upper end.

  LOGIC CODE LINES (ONLY logic code files — NOT boilerplate):
    INCLUDE: .py .js .ts .jsx .tsx .go .rs .java .cs .cpp .c .sh .bash .rb .sql .kt .swift and similar
    EXCLUDE: .html .css .json .md .yaml .yml .csv .xml .toml .lock and all other markup/data/template files
    AI generates boilerplate cheaply; a human expert would write a template, not hand-author every line.
    Expert writes 80–130 LoC/hr of logic code. 100 lines = ~1h, 300 lines = 2.5h, 800 lines = 6h.

  READ/SEARCH CALLS (investigation and analysis):
    File reads + grep/glob/search calls indicate exploration and research effort even with no code output.
    A session with 40+ reads required skilled navigation and judgment. Add 0.5–1h for read-heavy sessions.

  TOOL INVOCATIONS (non-coding work and execution breadth):
    Total tool calls are the primary signal for tasks producing no code output: document synthesis,
    image analysis, browser automation, presentation work, data processing. When lines_h ≈ 0 and
    reads_h is low, tool_invocations tells you how much the AI was actually doing.
    10 tools ≈ +0.25h  |  50 tools ≈ +0.40h  |  100 tools ≈ +0.50h  |  200 tools ≈ +0.55h

  Use the HIGHEST of (active_h, turns_h, lines_h, tools-based estimate) as your base,
  then add reads and tools contributions. Apply upward judgment for complex, decision-heavy work.

STEP 3 — Apply complexity adjustments based on what you read in the transcript:
  Iteration and rework (many turns on same files, errors corrected)  → +25–50% on base
  Broad scope (10+ files, multiple systems)                          → +20–30% for coordination overhead
  Design decisions with lasting impact (architecture, APIs, schemas) → scale toward upper anchors
  Novel problem-solving (not routine, required creative synthesis)   → scale toward upper anchors
  Pure execution (copying files, running scripts, pushing code)      → do NOT inflate; cap at 0.5h

STEP 4 — Reality-check against outcome anchors:
  0.25h    — Trivial: install a package, run a CLI command, toggle a config, push to git
  0.5h     — Simple: minor code edit, format/style tweak, rename, answer a question
  1.0–1.5h — Moderate: implement a small feature, debug a known issue, draft a short document
  2–4h     — Substantial: design + implement a feature, write a detailed report, complex data analysis
  4–8h     — Major: architect a new module, build a complete tool, comprehensive multi-step research
  8–16h    — Large: build a full system from scratch, multi-day design-implement-test cycle

CAPS AND FLOORS:
